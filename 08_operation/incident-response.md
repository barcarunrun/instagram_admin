# インシデント対応 (Incident Response)

## インシデント定義

### 重要度レベル

- **SEV-1 (Critical)**: サービス全体が利用不可
- **SEV-2 (High)**: サービスの一部が利用不可
- **SEV-3 (Medium)**: 機能が正常に動作していない
- **SEV-4 (Low)**: 軽微な問題

## インシデント報告フロー

1. 問題検出
2. インシデント作成 (INCIDENT-XXX)
3. チャネルに通知 (#incidents チャネル)
4. リーダー決定
5. 対応開始

## インシデント対応手順

### ステップ 1: 状況把握

```bash
# ステータス確認
npm run health-check

# ログ確認
npm run logs | tail -100

# メトリクス確認
curl http://localhost:9090/metrics
```

### ステップ 2: 一時対応

- サービスの部分的な停止
- トラフィック制限
- キャッシュクリア

### ステップ 3: 根本原因調査

```bash
# ログ分析
grep "ERROR" logs/error.log

# データベース確認
npm run db:status

# 外部API確認
npm run check:external-apis
```

### ステップ 4: 修正実施

- コード修正
- ホットフィックス適用
- ロールバック検討

### ステップ 5: 復旧確認

```bash
npm run health-check
npm run test:e2e
```

## ポストモーテム

インシデント解決後、以下を実施:

1. ポストモーテム会議実施 (24 時間以内)
2. 根本原因分析
3. 再発防止策を定義
4. 知見を共有
