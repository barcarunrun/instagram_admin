# ADR-003: 認証・認可ポリシー

## ステータス

提案中

## 背景

ユーザー認証と認可は、セキュリティの基本である。

ユーザーの個人情報を保護し、不正なアクセスを防ぐために、適切な認証・認可方式を選定する必要がある。

## 決定

### 認証方式

- **JWT (JSON Web Token)** を使用
- 認証後、ユーザーには署名付き JWT トークンを発行
- トークンの有効期限: 1 時間 (Access Token)
- リフレッシュトークン: 7 日間

### パスワード保存

- **bcrypt** を使用してハッシュ化
- ハッシュアルゴリズム: bcrypt (cost: 10)
- ソルト: 自動生成

### 認可方式

- **Role-Based Access Control (RBAC)**
- ロール: admin, moderator, user
- リソースごとに必要なロールを定義

### 多要素認証 (MFA)

- 将来的にサポート予定 (Email OTP, TOTP)
- 初期実装では不要

## 理由

### JWT の選択

- **利点**
  - ステートレスで スケーラブル
  - モバイルアプリとの相性が良い
  - 署名により改ざん検出可能

- **欠点**
  - トークン失効の管理が複雑
  - トークンサイズが大きい

### bcrypt の選択

- **利点**
  - 計算コストが高く、ブルートフォース攻撃に強い
  - Python/Node.js など多くの言語で実装されている

- **欠点**
  - 計算時間がかかる

### RBAC の選択

- **利点**
  - シンプルで理解しやすい
  - 多くの Web フレームワークでサポート

- **欠点**
  - 複雑な権限管理には向かない

## 代替案

### 代替案 1: OAuth 2.0

- **利点**: ユーザーパスワード管理が不要、ソーシャルログイン対応
- **欠点**: 実装が複雑

### 代替案 2: Session ベース認証

- **利点**: シンプル、サーバー側で制御可能
- **欠点**: スケーリングが難しい、CSRF 対策必要

### 代替案 3: Attribute-Based Access Control (ABAC)

- **利点**: 柔軟な権限管理
- **欠点**: 複雑性が高い

## 実装仕様

### トークン内容

```json
{
  "sub": "user_id",
  "email": "user@example.com",
  "role": "user",
  "iat": 1234567890,
  "exp": 1234571490
}
```

### API エンドポイント

- `POST /api/auth/register`: ユーザー登録
- `POST /api/auth/login`: ログイン
- `POST /api/auth/refresh`: トークン更新
- `POST /api/auth/logout`: ログアウト

## 影響

### ポジティブな影響

- セキュアなユーザー認証が実現
- スケーラブルな設計
- 将来の機能拡張が容易

### ネガティブな影響

- トークン失効管理の複雑性
- ログアウト後の即座の無効化が難しい

## リスク

1. **トークン盗難**: トークンを盗まれた場合
   - 対策: HTTPS 必須、トークン有効期限を短く

2. **パスワード強度**: 弱いパスワードの使用
   - 対策: パスワード要件の設定、バリデーション

3. **JWT シークレット漏洩**: 署名キーの漏洩
   - 対策: 環境変数で管理、定期的なキーローテーション
